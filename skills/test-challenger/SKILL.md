# Test Challenger

**Core principle**: A test is only as valuable as its independence from the code it tests. When the test's expected behavior was derived by observing the code rather than understanding the specification, the test cannot catch bugs — it can only catch regressions. This skill systematically identifies tests that are likely validating *what the code does* rather than *what the code should do*.

---

## The Oracle Problem

Every meaningful test has two components:
1. **Stimulus** — calling the code under test with specific inputs
2. **Oracle** — the expected result, derived from an independent source of truth

When a human writes a test, the oracle typically comes from a specification, domain knowledge, or a requirements document. When an AI generates a test by observing code, the oracle comes from the code itself. This creates a circular dependency: the test validates the code, but the code defined the test. The test becomes a tautology — it can never fail for the right reasons.

This skill applies a structured set of **challenge patterns** to each test to estimate the probability that its oracle was derived from the code rather than from genuine domain understanding.

---

## How to Execute This Skill

### STEP 1 — Gather Context

You need two things:
1. **The test file(s)** — the tests being challenged
2. **The source file(s)** — the code being tested

If only tests are provided, ask for the source. If only source is provided, ask for the tests. You need both to perform the analysis.

Optionally useful (ask if available):
- **Specification, requirements, or documentation** for the code under test — this is the independent oracle you'll compare against
- **Git history or PR context** — knowing the tests were AI-generated increases prior suspicion
- **The language/framework** — some patterns are framework-specific

### STEP 2 — Classify Each Test

For every test case, work through the **False Positive Pattern Catalog** below. A single test may exhibit multiple patterns. Assign each test a **Challenge Verdict**:

| Verdict | Meaning |
|---------|---------|
| 🟢 **Solid** | Test has an independent oracle; assertions are meaningful and would catch real bugs |
| 🟡 **Suspicious** | Test exhibits 1-2 weak patterns; might be correct but warrants human review |
| 🔴 **Likely False Positive** | Test exhibits strong indicators that its oracle was derived from code, not from domain knowledge |
| ⚫ **Vacuous** | Test provides zero verification value — it will pass regardless of correctness |

### STEP 3 — Apply the False Positive Pattern Catalog

Work through these patterns for each test. Not every pattern applies to every test — skip irrelevant ones. The patterns are ordered from most obvious to most subtle.

---

## False Positive Pattern Catalog

### Pattern 1: Tautological Test 🔴
**What it is**: The test reimplements the function's logic in the assertion, making it structurally impossible to fail.

**How to detect**:
- The assertion mirrors the implementation step-by-step
- The expected value is computed using the same algorithm as the source
- The test essentially says "does `f(x)` equal `f(x)`?"

**Example**:
```python
# Source
def calculate_tax(amount, rate):
    return amount * rate * 0.01

# Tautological test — reimplements the same formula
def test_calculate_tax():
    result = calculate_tax(100, 20)
    assert result == 100 * 20 * 0.01  # ← just repeating the code
```

**Challenge question**: "If the implementation had a bug (e.g., `rate * 0.1` instead of `rate * 0.01`), would this test catch it?" If the answer is no because you'd also change the expected value, it's tautological.

---

### Pattern 2: Hardcoded Oracle From Execution 🔴
**What it is**: The expected value was obtained by running the code and copying the output, not by computing it from domain knowledge.

**How to detect**:
- Magic numbers in assertions with no explanatory comment
- Expected values that are suspiciously specific (many decimal places, odd structures)
- The expected value would only be known if you ran the code first
- No specification or formula justifies the expected value

**Example**:
```python
def test_compute_score():
    result = compute_score(users=[...], weights=[...])
    assert result == 73.4821  # ← where did this number come from?
```

**Challenge question**: "Could a developer who has never seen the implementation derive this expected value from the requirements alone?" If not, the oracle likely came from execution.

---

### Pattern 3: Snapshot/Approval Test Without Specification 🟡
**What it is**: The test captures a snapshot of current behavior and asserts against it. This is legitimate for regression detection, but has zero value for correctness validation.

**How to detect**:
- `assertSnapshot`, `toMatchSnapshot`, `toMatchInlineSnapshot`, or similar
- Large expected outputs that look like they were copy-pasted from actual output
- Tests named `should_match_expected_output` without specifying what "expected" means

**Challenge question**: "Was this snapshot reviewed against a specification before being committed, or was it auto-accepted?" If auto-accepted, it's locking in current behavior — bugs included.

---

### Pattern 4: Assertion-Free or Weak-Assertion Test ⚫
**What it is**: The test executes the code but asserts nothing meaningful — or asserts something trivially true.

**How to detect**:
- No assertions at all (only calls the function — "smoke test")
- Only asserts non-null / non-empty / no-exception
- Asserts type but not value (`assertIsInstance(result, dict)`)
- Asserts length but not content

**Example**:
```python
def test_process_data():
    result = process_data(input_data)
    assert result is not None  # ← this tells us almost nothing
    assert len(result) > 0     # ← nor does this
```

**Challenge question**: "What specific wrong behavior would this test catch?" If you can't name one, it's vacuous.

---

### Pattern 5: Happy-Path-Only Coverage 🟡
**What it is**: Tests only exercise the code paths that already work, never probing boundaries, edge cases, or error paths where bugs typically live.

**How to detect**:
- All test inputs are "nice" values (small positive integers, short strings, non-empty lists)
- No tests for: null/empty inputs, boundary values, maximum sizes, negative numbers, Unicode, concurrent access, error conditions
- The test suite covers lines but not decision boundaries

**Challenge question**: "What would happen if I fed this function malformed, empty, enormous, or adversarial input? Is there a test for that?" If not, the coverage is cosmetic.

---

### Pattern 6: Over-Mocked Test 🟡
**What it is**: So many dependencies are mocked that the test only verifies internal wiring, not actual behavior. The test passes regardless of whether the real dependencies work correctly.

**How to detect**:
- More mock setup lines than actual test logic
- Mocks return hardcoded values that match what the code expects
- The test would pass even if the real dependency was completely broken
- Mock verification checks that a method was called, not that the result is correct

**Example**:
```python
def test_send_notification():
    mock_email = Mock()
    mock_email.send.return_value = True
    service = NotificationService(email=mock_email)
    result = service.notify(user, message)
    mock_email.send.assert_called_once()  # ← only checks wiring
    assert result is True                  # ← always true because mock returns True
```

**Challenge question**: "If I replaced the mock with a real (broken) implementation, would this test still pass?" If yes, it's testing the mocks, not the code.

---

### Pattern 7: Implementation-Coupled Assertion 🔴
**What it is**: The test asserts on implementation details (internal state, call order, private methods) rather than observable behavior. If the implementation is refactored without changing behavior, the test breaks — but if the behavior has a bug, the test might not catch it.

**How to detect**:
- Assertions on private/internal method calls
- Assertions on specific execution order when order doesn't matter
- Assertions on internal data structures rather than output
- Tests that break on refactoring without behavior change

**Challenge question**: "Could I refactor the implementation (without changing external behavior) and would this test break?" If yes, it's coupled to implementation, not behavior.

---

### Pattern 8: Coincidental Pass 🔴
**What it is**: The assertion passes by coincidence — the expected and actual values happen to match for the wrong reasons.

**How to detect**:
- Assertions on generic values that many incorrect implementations would also produce (0, 1, empty list, true)
- Only one input tested, and it happens to be a fixed point
- The test would pass with a trivially wrong implementation (e.g., `return 0` passes a test that expects 0 for a specific input)

**Example**:
```python
def test_multiply():
    assert multiply(0, 5) == 0    # ← passes even if multiply() just returns 0
    assert multiply(1, 7) == 7    # ← passes even if multiply() just returns the second arg
```

**Challenge question**: "What is the simplest wrong implementation that would still pass this test?" If a trivially wrong implementation passes, the test has low discriminative power.

---

### Pattern 9: Missing Negative Test 🟡
**What it is**: The test suite only verifies that correct inputs produce correct outputs, but never verifies that incorrect inputs are rejected or handled properly.

**How to detect**:
- No tests for invalid input (wrong types, out-of-range values, malformed data)
- No tests for error conditions the code claims to handle
- No tests verifying that exceptions are raised when they should be
- Code has error handling branches with zero test coverage

**Challenge question**: "Does this code have error handling? Are there tests proving that error handling works correctly?"

---

### Pattern 10: Specification Blindness (Meta-Pattern) 🔴
**What it is**: The umbrella pattern — across the entire test suite, there is no evidence that any test was informed by a specification, requirement, or domain rule. All tests appear to be derived from code observation.

**How to detect**:
- Test names describe implementation (`test_returns_map_with_keys`) not behavior (`test_discounted_price_applies_to_premium_members`)
- No domain language in test descriptions
- Expected values have no explanatory comments connecting them to business rules
- The test file reads like a mechanical trace of the code, not an independent behavioral contract

**Challenge question**: "If I deleted the source code and only had these tests + a specification, could I rewrite the code correctly?" If the tests don't encode enough domain knowledge, they're mirrors, not contracts.

## STEP 4 — Produce the Challenge Report

For each test (or logical group of tests), produce:

```
### [Test Name / Description]
**Verdict**: 🟢/🟡/🔴/⚫
**Patterns detected**: [list of pattern numbers and names]
**Reasoning**: [1-3 sentences explaining WHY this test is suspicious]

```

Then produce a **Summary**:

```
- ⚫ Vacuous: N (XX%)

[Top 3 most concerning tests with brief reasoning]

## Recommendations
[Prioritized list of actions to improve the test suite]
```

---

## STEP 5 — Suggest Remediation

For each suspicious or false-positive test, suggest one of:

1. **Rewrite with independent oracle** — derive expected values from specification/requirements, not from code execution
2. **Add boundary/edge case variants** — keep the existing test but add tests that probe where bugs actually hide
3. **Replace with property-based test** — instead of checking specific outputs, check invariants that must hold for any correct implementation
4. **Delete and replace** — when the test is vacuous or tautological, removing it is better than keeping false confidence
5. **Add specification comment** — if the test IS correct, add a comment explaining WHY the expected value is correct, linking to a requirement or domain rule

---

## Calibration Notes

Not every AI-generated test is a false positive. Some tests will be perfectly fine — especially when:
- The behavior being tested is simple and obviously correct
- The test checks a well-known invariant (e.g., sorting produces sorted output)
- The test was generated from a clear docstring or type signature that encodes the spec
- The test exercises error handling paths that are independently understandable

The goal isn't to reject all generated tests — it's to flag the ones where the oracle is suspect and human domain knowledge is needed to validate correctness.

---

## Edge Cases and Judgment Calls

**"But all my tests were generated and they increased coverage from 30% to 80%!"**
Coverage measures which lines execute, not whether assertions are meaningful. A test with no assertions achieves coverage. A test with wrong assertions achieves coverage. Challenge each test independently — the coverage number is not evidence of quality.

**"The test name says 'should calculate correct tax' — doesn't that mean it's correct?"**
The name reflects intent, not reality. If the expected value `20.0` in the assertion was derived from running `calculate_tax(100, 20)` rather than from knowing that 20% of $100 is $20, the name is aspirational, not factual.

**"What about regression tests? Aren't those meant to mirror current behavior?"**
Yes — but they should be explicitly labeled as regression/snapshot tests, not as correctness tests. If a test was generated to lock in current behavior, it's only valuable if someone first verified that current behavior is correct. If not, it's locking in bugs.
